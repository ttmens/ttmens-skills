# Idea-to-Product Workflow

将想法从概念推进到可交付产品的完整工作流。适用于代码类产品，也适用于技能、工具、文档等产品形态。

## 总览

```
💡 想法 → 🔬 深度调研 → 🎨 产品设计 → 🔥 Grill 评审 → 📋 实现计划 → ⚙️ 执行 → 📦 交付
(discuss)  (research)    (design)     (challenge)   (plan)      (build)   (ship)
```

**核心原则：**
- 调研先于设计 —— 不了解业界实践就设计，大概率是重新发明轮子
- Grill 先于实现 —— 方案没被质疑过，就不值得构建
- 每次迭代都要让用户看到可交互的东西，而不是抽象描述

## 什么时候用

- 用户有一个想法，想端到端构建出来
- 用户说"帮我做 X"、"建一个 Y"
- 功能需求，需要先验证再实现
- **任何需要先调研再设计的产品类任务**（不限于代码）

---

## Phase 1: 想法澄清与范围定义

**目标：** 搞清楚用户要什么、为谁做、最小可用版本是什么。

### 操作

1. **澄清问题：**
   - 解决什么问题？
   - 目标用户是谁？
   - 最小可用版本长什么样？

2. **定义范围边界：**
   - Must-have vs nice-to-have
   - 技术/资源约束
   - 时间预期

3. **输出：** 1-2 段清晰描述，说明我们要做什么。

### 转入 Phase 2

问用户："要不要先做一轮深度调研，看看业界怎么做的？还是你对方案已经有明确想法，直接开始设计？"

---

## Phase 2: 深度调研（新增）

**目标：** 在动手设计之前，充分调研业界最佳实践，避免闭门造车。

### 什么时候必须做深度调研

- 你不完全了解这个领域
- 用户说"调研一下"、"看看业界怎么做"
- 涉及开源工具选型、工作流设计、产品方案
- 初稿被用户否定（说明你的理解不够深入）

### 调研方法

**1. 多源搜索，不要只搜一个地方：**
```
GitHub 搜索 → 找到相关项目，看 stars、最近更新、issue 活跃度
npm/pip/crates → 找到相关包，看版本历史、依赖
技术博客/论坛 → 找到实践经验总结
```

**2. 不只读 README，要读关键实现文件：**
```
README.md → 了解项目定位和核心能力
源码中的核心文件 → 了解实际怎么做，而不只是声称怎么做
测试/示例文件 → 了解真实使用场景
```

**3. 提炼可复用的模式，而不只是罗列项目：**
```
❌ 坏输出：A 项目做了 X，B 项目做了 Y，C 项目做了 Z
✅ 好输出：业界有三种模式：模式 A（适合场景 1）、模式 B（适合场景 2）...
```

### 调研输出

```markdown
## 🔍 调研发现

### 业界模式
| 模式 | 代表项目 | 适用场景 | 优点 | 缺点 |
|------|---------|---------|------|------|
| ... | ... | ... | ... | ... |

### 核心洞察
1. ...（基于调研发现的关键结论）
2. ...

### 对我们的启示
- 应该采用 XX 模式，因为...
- 应该避免 YY，因为...
```

### 转入 Phase 3

展示调研结果，结合用户需求，提出设计方案方向。问用户："按这个方向设计可以吗？"

---

## Phase 3: 产品设计

**目标：** 基于调研和用户需求，设计出可落地的方案。

### 产品层面输出件

根据任务性质，输出件可能包含以下一个或多个：

| 输出件 | 说明 |
|--------|------|
| **PRD** | 产品需求文档：目标用户、问题陈述、功能清单、验收标准 |
| **竞品分析** | 对比 2-3 个竞品/开源方案的优劣势 |
| **架构设计** | 系统结构、模块划分、数据流 |
| **用户故事** | "作为 XX，我希望 XX，以便 XX" 格式 |
| **决策记录** | 为什么选 A 不选 B，记录决策理由 |
| **技能文档 (SKILL.md)** | 如果是 Hermes skill，包含触发条件、操作流程、陷阱、示例 |
| **模板文件** | 可复用的模板 |
| **脚本/工具** | 实现核心自动化能力的代码 |

### 设计要求

- **具体可执行**：不说"做一个待办管理"，而是"用 patch 操作 TODO.md 的 P0/P1/P2 分区"
- **有自动化能力**：不只是定义格式，而是能自动干活（解析、筛选、生成）
- **中文输出**：除非用户特别要求，否则所有面向用户的文档用简体中文

### 输出格式

设计文档保存到 `docs/plans/YYYY-MM-DD-产品名.md`，结构：

```markdown
# [产品名] 设计方案

> **For Hermes:** 用户确认后使用 subagent-driven-development skill 执行

## 背景
- 要解决的问题
- 目标用户

## 业界调研结论
- 最佳实践模式
- 我们的选择及理由

## 产品架构
- 模块划分
- 数据流

## 核心能力
| 能力 | 用户触发 | 系统行为 |
|------|---------|---------|
| ... | ... | ... |

## 输出件清单
- [ ] SKILL.md
- [ ] 模板文件
- [ ] 脚本/工具

## 文件结构
```
path/to/output/
├── ...
```
```

### 转入 Phase 4（Grill 评审）

**不要直接进入实现。** 先过 Grill 评审。

---

## Phase 4: Grill 评审（新增）

**目标：** 对设计方案进行批判性质疑，找出漏洞，确保方案值得构建。

### Grill 问题清单

对设计方案逐一拷问：

**1. 必要性拷问**
- 这个能力用户真的需要吗？还是我们自己想的？
- 有没有更简单的方式达到同样效果？
- 如果不做这个能力，用户会损失什么？

**2. 可行性拷问**
- 技术上有什么坑？刚才调研到的项目踩过什么坑？
- 依赖的工具有没有不稳定/已废弃的风险？
- 用户的实际环境能不能跑起来？

**3. 完整性拷问**
- 用户说了一句话，我们覆盖了所有隐含场景吗？
- 异常/边界情况处理了吗？
- 如果用户的数据格式和预期不一样怎么办？

**4. 体验拷问**
- 用户用起来需要几步？能不能更少？
- 输出对用户来说有意义吗？还是只是"格式对了但没用"？
- 初稿被用户否定的风险在哪里？

### Grill 输出

```markdown
## 🔥 Grill 结果

### 通过的项
- ...（设计合理，不需要修改）

### 需要修改的项
- [ ] ...（具体问题）→ 修改方案：...
- [ ] ...

### 风险项
- ⚠️ ...（需要用户确认或后续验证）
```

### Grill 结果处理

- **全部通过** → 转入 Phase 5（实现计划）
- **有修改项** → 修改设计方案 → 重新 Grill（最多 2 轮）
- **有重大风险** → 向用户汇报，让用户决定：继续 / 调整 / 放弃

### 转入 Phase 5

问用户："Grill 完成，方案已修改。可以开始实现了吗？"

---

## Phase 5: 实现计划

**Skill to load:** `writing-plans`

**目标：** 将设计方案拆解为可执行的 bite-sized 任务。

### 操作

1. 加载 `writing-plans` skill
2. 探索代码库/文件系统（如适用）
3. 设计架构
4. 写任务，每个任务包含：
   - 精确文件路径
   - 完整代码示例
   - 测试用例（TDD）
   - 验证步骤
5. 保存到 `docs/plans/YYYY-MM-DD-产品名.md`

### 转入 Phase 6

展示计划给用户。问："计划 OK 吗？开始执行？"

---

## Phase 6: 执行

**Skill to load:** `subagent-driven-development`

**目标：** 按计划执行，每个任务有审查。

### 操作

1. 加载 `subagent-driven-development` skill
2. 解析计划为 todo 列表
3. 对每个任务：
   - 分派执行 subagent
   - 规格审查
   - 代码质量审查
   - 修复问题
   - 标记完成

### 转入 Phase 7

所有任务完成 → 最终验证

---

## Phase 7: 验证与交付

**Skill to load:** `requesting-code-review`

**目标：** 最终安全/质量检查，交付给用户。

### 操作

1. 加载 `requesting-code-review` skill
2. 运行静态扫描
3. 运行测试
4. 独立 review
5. 自动修复循环（最多 2 轮）
6. 提交 + 推送到 Git

### 输出

- 验证报告
- 代码已提交/推送
- 产品可交付

---

## 快速开始

用户说"做 X"或"搞一个 Y"时：

1. **澄清**想法（Phase 1）
2. 问："走哪个流程？"
   - **全流程** — 所有 7 个阶段
   - **调研 + 设计** — 只做 Phase 2-3（适合方案设计）
   - **调研 + 设计 + Grill** — Phase 2-4（最常用）
   - **直接实现** — 从 Phase 5 开始（用户已有明确方案）

3. 按阶段推进，每步展示输出

## 跳过阶段指南

有些情况可以跳过：

- **跳过调研**：技术成熟、方案明确、用户已有清晰方向
- **跳过 Grill**：任务极简单（< 15 分钟），且没有设计取舍
- **跳过 Spike**：技术风险低，不需要原型验证
- **跳过详细计划**：任务极简单

但 **永远不要跳过验证**（Phase 7）—— 交付给用户的东西必须经过检查。

## 错误恢复

### 调研后发现此路不通
- 向用户报告发现
- 建议替代方案
- 让用户决定：转向 / 放弃

### Grill 发现方案有根本问题
- 向用户汇报
- 建议重新设计或放弃
- 如果用户坚持继续，记录风险并执行

### 实现中遇到调研没覆盖的坑
- 停下来，补充调研
- 更新设计方案
- 重新 Grill 修改部分

## 整合点

| Phase | 加载 Skill | 关键输出 |
|-------|-----------|----------|
| 1. 想法澄清 | (无) | 需求定义 |
| 2. 深度调研 | (无，用 web/search 工具) | 调研报告 + 核心洞察 |
| 3. 产品设计 | (无) | 设计方案 + 产品输出件 |
| 4. Grill 评审 | (无) | Grill 结果 + 修改方案 |
| 5. 实现计划 | `writing-plans` | 实现计划文档 |
| 6. 执行 | `subagent-driven-development` | 可工作的产品 |
| 7. 验证交付 | `requesting-code-review` | 已验证的交付物 |

## 记住

```
调研先于设计
Grill 先于实现
验证先于交付
每次迭代都要让用户看到东西
```
